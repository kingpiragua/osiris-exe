# 15 — Roadmap

## Purpose

To track where OSIRIS.EXE is going — versions, milestones, and priorities — at a
glance.

## Scope

Version history and forward plan for the `osiris-exe` application and the
franchise. Not detailed task tracking.

## Principles

- Direction over deadlines; sequence reflects what deepens the experience most.
- Each milestone has a clear intent, not just a feature list.
- Kept current; this is the canonical "what's next."

## Current Status

**Shipped (osiris-exe):**
- v0.1–0.2 — CRT boot sequence; sacred archive aesthetic.
- v0.3 — boot → interactive terminal handoff.
- v0.4 — access ritual; routing to the first recovered memory.
- v0.5 — single-cursor lifecycle; DISK recovery arc.
- v0.6 — Recovered Memory 001 record → cut-to-black → playback shell.
- v0.7 — return-to-archive, persistence, memory-page hum, degradation texture;
  hidden `reset`.
- v0.8 — content model + dynamic routing: recovered memories are data
  (`src/content/memories/`), rendered via `/archive/[id]`. Adding a memory is a
  data file, not a redesign. (Master Blueprint documentation foundation added.)

**Now:** Authoring the next recovered memory and/or real panel artwork.

## Open Questions

- v0.8 priorities: real Panel 001 artwork vs. content/data model vs. dynamic
  archive routing.
- When to externalize content (Content System) before adding more memories.
