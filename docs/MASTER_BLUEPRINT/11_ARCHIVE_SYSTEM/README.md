# 11 — Archive System

## Purpose

To define the archive itself — how artifacts are catalogued, addressed, and
accessed through ritual.

## Scope

Artifact taxonomy (recovered memories, reports, transmissions, audio, images),
identifiers and routing, the access ritual, recovery state/persistence, and
"return to archive." Distinct from a single memory's playback (Motion Comic
Engine).

## Principles

- The archive *contains* many artifact types; a recovered memory is one kind.
- Access is granted through ritual, never a menu.
- Recovery is remembered (persistence) so returning feels acknowledged.
- Language is fixed: recover / recovered memory / memory playback / return to
  archive.

## Current Status

Implemented for the first artifact: `/archive/001` (Recovered Memory 001) with
record → cut-to-black → playback, recovery persistence, and Esc-to-return.

## Open Questions

- Identifier scheme and dynamic routing as artifacts multiply.
- How non-memory artifacts (reports, audio) are accessed and presented.
- Whether an in-world "index" of recovered artifacts exists, and how.
